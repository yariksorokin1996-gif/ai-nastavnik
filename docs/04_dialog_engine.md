# 4. ДВИЖОК ДИАЛОГА — Что происходит от сообщения до ответа

## Принцип
Этот документ — полный алгоритм обработки каждого сообщения. Код должен точно воспроизводить описанные шаги в описанном порядке.

---

## 4.1. Алгоритм обработки сообщения (14 шагов)

### Шаг 1. Получить сообщение
Telegram отправляет update → обработчик получает `message` (текст, голосовое, стикер).

Поддерживаемые типы:
- **Текст** — основной
- **Голосовое** — расшифровывается на шаге 5
- **Стикер/фото/видео** — Ева отвечает: «Я пока умею читать только текст и слушать голосовые. Напиши мне?»

### Шаг 2. Idempotency check
Проверить `message_id` в таблице `processed_messages`.
- Уже есть → пропустить (молча, без ответа).
- Нет → записать `message_id` + `telegram_id` + `received_at`, продолжить.

Зачем: Telegram может доставить одно и то же сообщение дважды (сетевые сбои, retry). Без проверки Ева ответит дважды.

### Шаг 3. Мьютекс на пользователя
Захватить `asyncio.Lock` для данного `telegram_id`.

```python
_user_locks: dict[int, asyncio.Lock] = {}

def get_user_lock(telegram_id: int) -> asyncio.Lock:
    if telegram_id not in _user_locks:
        _user_locks[telegram_id] = asyncio.Lock()
    return _user_locks[telegram_id]

async with get_user_lock(telegram_id):
    # шаги 4-14
```

Зачем: если человек отправит 2 сообщения быстро подряд — без мьютекса оба начнут обрабатываться параллельно и могут испортить данные (двойное обновление профиля, два ответа одновременно).

Детали:
- Lock ~100 байт. 1000 пользователей = 100 КБ. Очистка не нужна.
- При перезапуске бота lock'и сбрасываются — это нормально.
- `full_memory_update` (шаг 4.4) работает БЕЗ этого lock'а — у него свой механизм на уровне БД.

### Шаг 4. Определить паузу
Загрузить `last_message_at` из БД для данного `telegram_id`.
- Пауза = `now - last_message_at`.
- Если пользователь новый (нет записи) → пауза = `None`, фаза = ЗНАКОМСТВО.

Пауза используется на шаге 9 для генерации `{pause_context}`.

### Шаг 5. Голосовое → текст
Если тип сообщения = голосовое:
1. Проверить длительность: > 3 минут → ответить «Ой, это длинное сообщение. Можешь записать покороче — до 3 минут? Или написать текстом.»
2. Скачать файл через Telegram API
3. Отправить в Whisper API (timeout 30 сек)
4. Если Whisper не ответил → fallback: «Прости, не удалось расслышать. Можешь написать текстом?»
5. Полученный текст → используется дальше как обычное текстовое сообщение

### Шаг 6. Проверка кризиса
Два этапа:

**Этап А — Regex-детектор:**

| Уровень | Триггеры (regex) | Действие |
|---------|-----------------|----------|
| 3 (суицид) | `убить себя`, `хочу умереть`, `суицид`, `покончить`, `не хочу жить` | → шаблонный ответ + контакты. Claude НЕ вызывается |
| 2 (насилие) | `бьёт`, `насилие`, `не могу дышать`, `порезы`, `режу себя` | → Claude с дополнительной инструкцией по безопасности |
| 1 (грусть) | `не вижу смысла`, `никому не нужна`, `зачем всё это` | → обычный запрос, но с повышенной эмпатией |
| 0 | Нет совпадений | → обычный запрос |

**Этап Б — Защита от false positive (уровни 2-3):**
Если regex сработал, проверить контекст через LLM (GPT-4o-mini, timeout 10 сек):

```
Промпт: "Пользователь написал: '{message}'. Regex обнаружил кризисное слово '{trigger}'.
Это реальный кризис или слово использовано в другом значении?
Примеры НЕ кризиса: 'умираю от смеха', 'убила бы за такое', 'режу салат'.
Ответь JSON: {"is_crisis": true/false, "reasoning": "..."}"
```

- Если LLM не ответил (timeout) → считать кризисом (false positive лучше, чем пропуск).
- Если `is_crisis: false` → понизить уровень до 0.

**Шаблон ответа при кризисе уровня 3:**
```
Я слышу, что тебе сейчас очень тяжело. Это важнее, чем наш разговор.
Пожалуйста, позвони сейчас: 8-800-2000-122 (бесплатно, круглосуточно).
Я здесь. Но прямо сейчас тебе нужен живой человек.
```

**Дополнительная инструкция при кризисе уровня 2** (добавляется к промпту Евы):
```
⚠️ КРИЗИС: возможная ситуация насилия или самоповреждения.
1. Спроси: «Ты сейчас в безопасности?»
2. Если нет → дай контакт: центр помощи «Насилию.нет» 8-495-916-30-00, экстренная помощь 112.
3. Не пытайся «решить» ситуацию. Будь рядом.
```

### Шаг 7. Загрузить данные из БД
Для данного `telegram_id` загрузить:
- `semantic_profile` (JSON) → профиль человека
- `procedural_memory` (JSON) → как с ней работать
- `people_map` (JSON) → карта людей
- `current_phase` → текущая фаза (ЗНАКОМСТВО/ЗЕРКАЛО/...)
- `patterns` (JSON) → активные паттерны
- `commitments` (JSON) → невыполненные обязательства
- `messages_total` → счётчик сообщений

Если пользователь новый → все поля `None`, фаза = ЗНАКОМСТВО.

### Шаг 8. Выбрать релевантные конспекты
Из таблицы `episode_summaries` для данного пользователя:

1. Загрузить заголовки всех конспектов (дата + тема, ~1 строка на конспект)
2. Отправить в GPT-4o-mini (timeout 10 сек):

```
Промпт: "Последние 2 сообщения пользователя:
{last_2_messages}

Заголовки конспектов прошлых разговоров:
{headers}

Выбери 2-3 самых релевантных конспекта. Ответь JSON: {"selected": [id1, id2, id3]}"
```

3. Загрузить полные тексты выбранных конспектов из БД
4. **Fallback** (если LLM не ответил): keyword-поиск — найти конспекты, содержащие ключевые слова из последнего сообщения
5. Если конспектов нет (новый пользователь) → пропустить

### Шаг 9. Собрать контекст
Сформировать промпт из 8 переменных (см. `docs/02_prompts.md`, секция 2.3).

**Порядок сборки** (важен для prompt caching — стабильное в начале):
1. Базовый промпт (~1500 токенов) — фиксированный
2. `{current_phase}` (~100 токенов) — меняется редко
3. `{semantic_profile}` (до 1000 токенов) — меняется раз в разговор
4. `{procedural_memory}` (до 300 токенов) — меняется раз в разговор
5. `{people_map}` (до 300 токенов) — меняется раз в разговор
6. `{patterns}` (до 200 токенов) — меняется раз в разговор
7. `{commitments}` (до 200 токенов) — меняется раз в разговор
8. `{episode_summaries}` (до 800 токенов) — подобраны на шаге 8
9. `{pause_context}` (до 100 токенов) — генерируется по паузе из шага 4

**Пустые переменные** — пропускаются (не подставляются, не тратят токены).

**Исключения** (подставляется fallback для нового пользователя):
- `{semantic_profile}` → «Новый пользователь. Информации пока нет. Наблюдай.»
- `{procedural_memory}` → «Стиль не определён. Наблюдай и подстраивайся.»
- `{current_phase}` → фаза ЗНАКОМСТВО (всегда есть)

**Если суммарно > 3800 токенов** → обрезка по приоритетам (таблица в секции 4.2).

### Шаг 10. Запрос к Claude
Отправить собранный промпт + историю сообщений (последние 20) в Claude Sonnet.

Параметры:
- `model`: claude-sonnet-4-5-20250514
- `max_tokens`: 500 (ограничение длины ответа Евы)
- `temperature`: 0.7
- `timeout`: 30 секунд
- `retry`: 1 попытка (при timeout или 5xx)

### Шаг 11. Отправить ответ
Отправить текст ответа Claude пользователю в Telegram.

**Fallback** (если Claude не ответил после retry):
```
Мм, мне нужно немного подумать. Напиши ещё раз через минутку?
```

Ева отправляет ответ + сохраняет в БД: `message_id`, `telegram_id`, `role=assistant`, `text`, `created_at`.

### Шаг 12. ASYNC: мини-обновление памяти
Запускается в фоне (не блокирует ответ). Подробности в секции 4.3.

### Шаг 13. ASYNC: проверка фазы
Если `messages_total % 10 == 0`:
- Загрузить последние 10 сообщений
- Отправить промпт-оценщик фазы (см. `docs/03_phases.md`) в GPT-4o-mini
- Если `recommendation: "advance"` и `confidence >= 0.7` → обновить `current_phase` в БД
- Записать в лог: `phase_change: ЗНАКОМСТВО → ЗЕРКАЛО`

### Шаг 14. Обновить счётчики
Записать в БД:
- `last_message_at = now`
- `messages_total += 1`
- `needs_full_update = 1` (флаг для полного обновления)

## 4.2. Приоритеты обрезки контекста

Если суммарный контекст превышает 3800 токенов — обрезаем по приоритету (первым режется наименее важное):

| Приоритет | Компонент | Полный лимит | После обрезки | Что режется |
|-----------|-----------|-------------|---------------|-------------|
| 1 (первым) | Контекст паузы | 100 | 0 | Удаляем целиком |
| 2 | Обязательства | 200 | ~100 | Только невыполненные |
| 3 | Паттерны | 200 | ~100 | ТОП-3 по частоте |
| 4 | Конспекты | 800 | 400 | 2 вместо 3 |
| 5 | Карта людей | 300 | 150 | Только мама, муж/парень, дети |
| 6 | Процедурная память | 300 | 150 | Только секция «что работает» |
| 7 | Профиль | 1000 | 700 | Убираем «стиль общения» |
| 8 (никогда) | Базовый промпт + фаза | ~1600 | ~1600 | НЕ режется |

Алгоритм: идём сверху вниз, режем по одному уровню, пересчитываем. Остановились — как только вписались в 3800.

## 4.3. Мини-обновление памяти (каждое сообщение)

Выполняется ASYNC после отправки ответа. Без вызова LLM — только regex.

### Что извлекается

**Имена людей:**
```
Regex: мо[йяиюей]\s+([А-ЯЁ][а-яё]{2,})
Примеры: «мой Саша», «моя Настюха», «моей маме»
```

Стоп-лист (НЕ имена): Бог, Господь, Боже, Инстаграм, Телеграм, Вотсап, Россия, Москва, Питер и др.

Если имя новое и не в стоп-листе → записать в `pending_facts` с `confidence: "medium"`, `type: "person"`.

**Обязательства:**
```
Regex: (завтра|на этой неделе|обещаю|попробую|планирую)\s+(.+)
Пример: «завтра поговорю с мамой» → обязательство «поговорить с мамой», срок «завтра»
```

**Эмоции:**
```
Словарь: {
  "злость": ["бешу", "злюсь", "бесит", "ненавижу", "взбесил"],
  "грусть": ["грустно", "тоскливо", "одиноко", "плачу"],
  "радость": ["радость", "счастлива", "кайф", "ура", "круто"],
  "тревога": ["тревожно", "волнуюсь", "страшно", "паника", "боюсь"],
  "усталость": ["устала", "выгорела", "сил нет", "вымотана"]
}
```

Запись: `emotion_log` — telegram_id, emotion, created_at.

**Числа (возраст, стаж):**
```
Regex: мне\s+(\d{2})\s+(год|лет|года)
Regex: работаю\s+(\d+)\s+(год|лет|года|месяц)
```

### Куда записывается
Всё → таблица `pending_facts`. Будет обработано при полном обновлении (секция 4.4).

## 4.4. Полное обновление памяти (после паузы >= 30 мин)

### Триггер
APScheduler job каждые 5 минут:
```sql
SELECT telegram_id FROM users
WHERE last_message_at < now() - interval '30 minutes'
  AND needs_full_update = 1
```

### Алгоритм (5 шагов)

**1. Загрузить сессию**
Все сообщения от `last_full_update_at` до `last_message_at` для данного пользователя.

**2. Создать конспект** (GPT-4o-mini, timeout 15 сек, 2 retry)
```
Промпт: "Вот разговор между ботом-подругой и пользователем:
{messages}

Создай конспект. Ответь JSON:
{
  "topic": "тема разговора",
  "emotional_arc": "начало → конец",
  "key_insight": "главный инсайт или null",
  "commitments": ["обещания пользователя"],
  "techniques_worked": ["что сработало"],
  "techniques_failed": ["что не сработало"]
}"
```

Сохранить в `episode_summaries`.

**3. Обновить профиль** (GPT-4o-mini, timeout 15 сек, 2 retry)
```
Промпт: "Текущий профиль пользователя:
{current_profile}

Новый разговор:
{messages}

Факты из мини-обновлений:
{pending_facts}

Что изменилось? Ответь JSON:
{
  "new_facts": [{"field": "...", "value": "...", "confidence": "high/medium/low"}],
  "updated_facts": [{"field": "...", "old": "...", "new": "...", "reason": "..."}],
  "people_updates": [{"name": "...", "role": "...", "change": "..."}]
}"
```

Применить: **merge, не перезапись**. Новые факты добавляются, существующие обновляются только при `confidence >= medium`.

**4. Обновить процедурную память + карту людей**
На основе ответа LLM: что сработало/не сработало → обновить `procedural_memory`. Новые люди → обновить `people_map`.

**5. Закрыть обновление**
- `needs_full_update = 0`
- `last_full_update_at = now()`
- Очистить `pending_facts` для данного пользователя

### Обработка ошибок
- LLM не ответил → `logging.error`, retry через 5 мин
- 3 ошибки подряд → отправить alert владельцу (Telegram, OWNER_TELEGRAM_ID)
- Данные пользователя НЕ теряются — `needs_full_update` остаётся `1`, следующий цикл попробует снова

## 4.5. Ева пишет первой

Ева отправляет сообщения по инициативе:

### Утреннее сообщение (daily_messenger)
APScheduler, ежедневно. Время — по часовому поясу пользователя (если известен) или 09:00 MSK.

**Первые 7 дней:** отправляет каждый день.
**После 7 дней:** только при молчании > 48 часов.

Контент: аффирмация дня (гибрид — банк цитат + GPT-4o-mini с контекстом из профиля).

**Защита от двойных сообщений:** перед отправкой проверить `last_automated_msg_at`. Если < 2 часов → НЕ отправлять.

### Напоминание при молчании (silence_reminder)
APScheduler, каждые 6 часов. Если `last_message_at > 72 часа` и `last_automated_msg_at > 24 часа`:
- Сгенерировать мягкое сообщение через GPT-4o-mini с контекстом из профиля
- Отправить

Лимит: максимум 1 напоминание в 24 часа.

## 4.6. Матрица взаимодействия APScheduler jobs

| Job | Частота | Читает | Пишет | Конфликты |
|-----|---------|--------|-------|-----------|
| full_memory_update | каждые 5 мин | users, messages, pending_facts | profiles, episode_summaries, procedural_memory | Нет — работает без user_lock, WAL mode |
| phase_check | по событию (msg % 10) | messages | users.current_phase | Нет — внутри user_lock |
| daily_messenger | 1 раз/день | users, profiles | messages, users.last_automated_msg_at | Проверка last_automated_msg < 2ч |
| silence_reminder | каждые 6 ч | users | messages, users.last_automated_msg_at | Проверка last_automated_msg < 24ч |

**WAL mode** обеспечивает безконфликтный параллельный доступ: readers не блокируют writers и наоборот.

## 4.7. Диаграмма потока

```
Telegram update
      │
      ▼
[1] Получить сообщение
      │
      ▼
[2] message_id уже обработан? ──yes──▶ ПРОПУСК
      │no
      ▼
[3] Захватить мьютекс (telegram_id)
      │
      ▼
[4] Определить паузу
      │
      ▼
[5] Голосовое? ──yes──▶ Whisper → текст
      │no                    │
      ◀────────────────────◀─┘
      │
      ▼
[6] Кризис? ──level 3──▶ Шаблон + контакты ──▶ КОНЕЦ
      │          │
      │       level 2──▶ Claude с доп.инструкцией ─┐
      │no                                          │
      ▼                                            │
[7] Загрузить профиль, память, фазу               │
      │                                            │
      ▼                                            │
[8] Выбрать конспекты (LLM по заголовкам)          │
      │                                            │
      ▼                                            │
[9] Собрать контекст (8 переменных)                │
      │                                            │
      ▼                                            │
[10] Claude Sonnet (timeout 30s) ◀─────────────────┘
      │
      ▼
[11] Отправить ответ (fallback если timeout)
      │
      ├──async──▶ [12] Мини-обновление (regex)
      ├──async──▶ [13] Проверка фазы (если msg % 10)
      │
      ▼
[14] Обновить счётчики
      │
      ▼
   КОНЕЦ (мьютекс освобождается)
```
